# Sanaka 从零开始的构建指南

这份文档是给第一次接触 `Sanaka` 的人看的。（文章编写作者是：GPT）

让你从一台空机器开始，逐步做到：

- 拉代码
- 安装依赖
- 本地跑起来
- 构建桌面应用
- 在需要时把 QEMU 嵌进打包结果

## 1. 先理解 Sanaka 是什么

`Sanaka` 是一个 Electron 桌面应用前端，本身不等于 QEMU。

你可以把它理解成两层：

1. `Sanaka` 负责界面、配置、控制台、运行时编排
2. `QEMU` 负责真正运行虚拟机。

所以构建时分两种情况：

- 只想开发和调试界面/逻辑：不用先处理 QEMU 打包、编译呀这些，电脑直接装上就可以。
- 想产出能直接分发的桌面应用：需要把 QEMU 一起嵌进去

## 2. 你需要准备什么

最基础的共通前置：

- Git
- Node.js
- npm

建议：

- Node.js 20 或更新的 LTS 版本

先确认环境：

```bash
node -v
npm -v
git --version
```

## 3. 拉代码

```bash
git clone https://github.com/steve372a/sanaka.git
cd sanaka
```

## 4. 安装依赖

```bash
npm install
```

如果你在国内网络环境下下载 Electron 很慢或失败，直接跑：

```bash
sh doctor
```

它会测速多个 npm / Electron 镜像，告诉你最快的是哪个，确认后再写入配置。

## 5. 先做最基本检查

这一步不启动 Electron，只检查前端和 TypeScript 构建是否过。

```bash
npm run build
```

如果这里只是想做类型检查：

```bash
npm run typecheck
```

如果想跑测试：

```bash
npm test
```

## 6. 本地直接运行 Sanaka

```bash
npm start
```

这个命令会先构建，再启动 Electron。

在 Linux 开发环境下，仓库现在会自动用 `--no-sandbox` 启动 Electron，避免 `chrome-sandbox` 的 `4755` 权限问题。

如果你只是开发前端界面：

```bash
npm run dev:ui
```

注意：

- `npm start` 是完整桌面应用体验
- `npm run dev:ui` 更适合只调网页界面

## 7. QEMU 在开发期怎么处理

当前仓库里，开发期默认依赖系统里的 QEMU。

也就是说：

- macOS：通常用你自己装好的 QEMU
- Windows：通常用你自己装好的 QEMU
- Linux：通常也先用系统已有的 QEMU

如果系统里没装 QEMU：

- 你仍然可以打开应用、编辑配置
- 但启动虚拟机会被禁用或报缺失

## 8. macOS 从零构建

### 8.1 先准备

你需要：

- 一台 macOS 机器
- 已完成 `npm install`
- 一份你自己编译好的 QEMU 构建目录

当前仓库常用脚本：

```bash
npm run pack:mac
```

它内部等价于：

```bash
bash ./scripts/package-sanaka-macos.sh /Volumes/sks/src/qemu-11.0.1/build-sanaka
```

也就是说，`npm run pack:mac` 这个快捷命令默认写死了一个本地 QEMU 构建路径。

如果你的 QEMU 不在这个路径，就不要直接用它，改用：

```bash
bash ./scripts/package-sanaka-macos.sh /你的/qemu/build目录
```

### 8.2 它会做什么

这个脚本会：

1. 执行 `npm run build`
2. 执行 `electron-builder --dir --mac`
3. 找到 `Sanaka.app`
4. 把你的 QEMU 嵌进去
5. 刷新 macOS 的应用注册

### 8.3 产物在哪里

通常在：

```bash
release/mac-arm64/Sanaka.app
```

### 8.4 更快的本地构建

如果你只是想快速得到 `.app` 测试包，可以看：

```bash
scripts/quick-build-macos-app.sh
```

## 9. Windows 从零构建

建议在真实 Windows 环境下做，不建议把正式 Windows 打包完全寄托在 WSL。

### 9.1 先准备

你需要：

- Windows
- Git for Windows
- Node.js
- npm

如果你已经有仓库副本，推荐先同步主分支并自动装依赖：

```bash
sh scripts/pull-main.sh
```

它会做：

- `git fetch origin`
- `git reset --hard origin/main`
- `npm install`
- `npm run build`

### 9.2 构建安装包

```bash
npm run pack:win
```

这会走 `electron-builder` 的 NSIS 安装包流程。

常见产物会在：

```text
release/
```

### 9.3 构建目录版并嵌入 QEMU

如果你更想先得到可运行目录，再把本机 QEMU 嵌进去：

```bash
npm run pack:win:dir
```

这个脚本会自动尝试查找：

- `C:\Program Files\qemu`
- `C:\Program Files\QEMU`

如果自动找不到，可以手动指定：

```bash
bash ./scripts/package-sanaka-windows.sh "C:/Program Files/qemu"
```

目录版产物通常在：

```text
release/win-unpacked
```

### 9.4 一个最短常用流程

```bash
sh scripts/pull-main.sh
npm run pack:win
```

## 10. Linux 从零构建

Linux 当前仓库里默认目标是 `deb`。

### 10.1 构建

```bash
npm run pack:linux
```

实际调用的是：

```bash
npx electron-builder --linux
```

### 10.2 产物

通常在：

```text
release/
```

### 10.3 当前 Linux 路线说明

当前仓库没有像 macOS / Windows 那样，单独写一套 Linux 内嵌 QEMU 的完整脚本流程。

也就是说当前默认思路更偏向：

- Linux 用户自行准备 QEMU
- Sanaka 负责前端与控制层
- 打包产物走 `deb`

## 11. 常见失败点

### 11.1 `npm install` 失败

常见原因：

- Electron 下载失败
- 网络/代理问题
- 本地 Node 版本太怪

先检查：

```bash
node -v
npm -v
```

然后再考虑镜像或代理。最省事的方式就是先跑：

```bash
sh doctor
```

### 11.2 `npm run build` 失败

优先看：

- TypeScript 报错
- 前端 AI 改坏类型
- 漏了新接口的 mock

这种通常不是环境问题，是代码本身有错。

### 11.3 macOS 打包失败

常见原因：

- QEMU 路径不对
- `codesign` 抽风
- 本地打包产物缓存脏了

当前仓库里，macOS 打包不是“纯 Electron 一条命令结束”，而是脚本二次处理，所以一定要把 QEMU 路径看准。

### 11.4 Windows 打包失败

常见原因：

- Electron 下载失败
- 杀毒软件/系统占用锁住 `Sanaka.exe`
- QEMU 目录没找到

如果报 `EBUSY`，优先怀疑：

- 已经有正在运行的 `Sanaka.exe`
- 目标目录里的旧产物被占用
