# 探讨管理

## 主题

探讨：

- 独裁
- 开放
- vibe coding
- 它们对 `Sanakaprix` 和 `Sanaka` 的影响

还有这几个现实问题：

1. 未来要不要逐渐退位，因为 vibe coding
2. 我作为 owner，会不会被架空
3. 我该如何管理 Sanaka，GitHub
4. 我该如何招募贡献者

## 先说结论

我觉得你现在最适合的，不是纯独裁，也不是完全开放，而是：

**创始人主导 + 明确规则 + 受控开放**

人话就是：

- 方向你定
- 最终拍板你来
- 别人可以贡献
- 但贡献不是来“接管”项目，而是来服务项目目标

这其实很接近开源世界常见的 founder-led / BDFL 路线，只是你要把“权力来自你本人”慢慢升级成“权力来自你写下来的规则”。

这样做的好处是：

- 你不会一开始就失去项目控制权
- 贡献者也不会觉得这是纯个人玩具，无法参与
- 项目长大以后，不会什么都卡在你一个人脑子里

## 一、独裁、开放、vibe coding 分别会怎么影响你

### 1. 独裁

独裁在开源里，不一定是坏词。更准确说法是创始人主导。

这类模式的优点很明显：

- 方向统一
- 设计语言不会被改烂
- 不容易被“看起来很合理、实际上把产品带偏”的建议拖走
- 决策速度快

对你这种有强审美、强产品直觉、强控制欲的人，其实很合适。

但问题也很明显：

- 你会成为瓶颈
- 你累了，项目就停
- 外部贡献者会不知道边界
- 所有人都要猜你在想什么

GitHub 的开源维护建议一直强调：维护者需要把流程、边界、接受标准写出来，不然随着项目受欢迎，维护者会越来越少写代码、越来越多处理沟通。  
参考：
- [GitHub Open Source Guide: Best practices for maintainers](https://github.com/github/opensource.guide/blob/main/_articles/best-practices.md)

所以，独裁不是不能用，而是不能停留在“全靠你临场判断”。

### 2. 开放

开放的优点：

- 能吸引更多人来试
- 能提高项目活力
- 能降低你一个人维护的压力

但开放的风险是：

- 低质量 PR 会很多
- 很多人贡献的不是你真正要的方向
- 讨论会膨胀
- 设计会碎

GitHub 和不少维护者经验都反复提一件事：  
不是没有人愿意贡献，而是“错误类型的贡献”太多，或者贡献路径不清楚。  
参考：
- [How open source maintainers keep contributors—and themselves—happy](https://github.com/readme/featured/contributor-relations)
- [How to manage casual contributors to open source projects](https://opensource.com/article/17/10/managing-casual-contributors)

所以“开放”不能等于“谁都能随便改主线”。

你需要的是：

- 贡献入口开放
- 决策权限不开放

### 3. vibe coding

vibe coding 对你这种项目的影响，非常大，而且是双面的。

好的地方：

- 你能更快试错
- 你不再完全受限于自己一个人的编码速度
- 你可以更快把产品脑子里的东西变成能运行的东西

坏的地方：

- 代码会更快膨胀
- 风格容易飘
- 责任边界会模糊
- “到底是谁决定的”会变得不清楚
- 你如果不盯方向，很容易被 AI 和贡献者一起推着跑

所以 vibe coding 不会自动让你该退位。  
它真正带来的，是一个新问题：

**你从“亲自写最多代码的人”，变成“定义方向、审查质量、收口决策的人”。**

这不是失权，而是角色升级。

## 二、未来要不要逐渐退位，因为 vibe coding

我的判断是：

**现在不该退。以后可以分权，但不该退位。**

原因很简单：

- `Sanaka` 现在的产品气质、方向、审美、优先级，基本都还是你在扛
- AI 能加速实现，但不能天然替代产品 owner
- 现在如果你退，项目很容易变成“别人也做了一堆功能，但不像 Sanaka”

真正应该变化的，不是你退不退，而是你从这三件事里慢慢退出一部分：

- 低价值重复实现
- 零散修补
- 所有 issue 都亲自回复

你应该保留的是：

- 路线图最终决定权
- 设计语言最终决定权
- 什么可以合并到主线的最终决定权

也就是说：

- 不退位
- 但逐步退出所有执行细节

## 三、你作为 owner，会不会被架空

会，如果你没有规则。  
不会，如果你把规则写清楚。

开源里 owner 被“架空”，通常不是别人突然夺权，而是下面这些情况慢慢发生：

- 你长期不响应
- 你没有明确治理规则
- 你没有写谁能决定什么
- 别人实际在干活，你只剩名义头衔
- GitHub 权限和事实权力脱节

研究和治理实践里反复出现一个点：  
维护者最容易变成瓶颈，也最容易在没有制度时被“事实上的活跃贡献者”替代。  
参考：
- [Governance.md Guide](https://governance.md/)
- [Governance in Practice: How Open Source Projects Define and Document Roles](https://arxiv.org/abs/2603.24879)

所以你要防的不是“别人太强”，而是“你没把 owner 身份制度化”。

### 你应该立刻写清楚的东西

- 项目 owner 是谁
- 谁有最终合并权
- 哪些改动必须先开 issue / proposal
- 哪些部分你保留最终审美和产品否决权
- 谁可以成为 maintainer，条件是什么
- maintainer 能做什么，不能做什么

如果这些不写，时间一长，谁最活跃谁就容易获得事实权力。

## 四、你该如何管理 Sanaka 和 GitHub

我建议你按四层来管。

### 第一层：方向

你来定：

- Sanaka 是什么，不是什么
- 当前版本优先级
- 体验底线
- 视觉语言底线
- 架构底线

这个要写进几个文件：

- `README.md`
- `context.md`
- `GOVERNANCE.md`
- `CONTRIBUTING.md`

### 第二层：权限

GitHub 不要一上来就把很多人都给写权限。

推荐顺序：

1. `issue / discussion / PR` 参与
2. triage 级别协作
3. 文档或测试维护权限
4. 某个子模块维护权限
5. 主仓库写权限

GitHub 也明确建议过：可以先邀请别人进组织，但不直接给 commit 权限，这是低压力参与方式。  
参考：
- [Elevating open source contributors to open source maintainers](https://github.blog/open-source/maintainers/elevating-open-source-contributors-to-open-source-maintainers/)

### 第三层：流程

你需要把项目变成“按流程协作”，而不是“看你当天心情决定”。

至少要有：

- issue 模板
- PR 模板
- bug / feature / design proposal 分类
- 标准标签
- 分支保护
- 必须 review 才能 merge

对 Sanaka 这种产品感很强的项目，我建议：

- 功能类改动：先 issue，再 PR
- 设计类改动：先提案，再实现
- 大改动：先写 spec

### 第四层：资产

你要把最重要的资产从“口头理解”变成“仓库内文件”。

包括：

- 产品原则
- 设计原则
- 架构约束
- 命名规则
- 机器包格式规则
- 运行时路线图
- 谁负责什么

不然 AI、贡献者、未来的你自己，都会不断偏航。

## 五、你该如何招募贡献者

先说现实：

开源项目不是“放到 GitHub 就有人来帮你写核心功能”。

大部分项目真正能吸引到的，都是：

- 顺手修 bug 的人
- 修文档的人
- 补测试的人
- 做小功能的人

所以不要幻想“我一开放，就有人帮我做完整系统”。

你真正应该做的，是设计一个让别人容易进来的贡献入口。

### 1. 明确你要招什么人

不要笼统写“欢迎贡献”。

你应该直接写：

- 需要前端界面协作者
- 需要 QEMU / runtime 方向协作者
- 需要文档整理者
- 需要模板与兼容性测试者
- 需要 Windows / Linux 打包验证者

具体，别人才能对号入座。

### 2. 先给低门槛任务

优秀维护者普遍会准备“容易开始”的入口任务。  
参考：
- [GitHub Docs: Contributing to open source](https://docs.github.com/en/enterprise-server%403.17/get-started/exploring-projects-on-github/contributing-to-open-source)
- [ONWARD maintainer framework](https://github.com/readme/guides/onward-maintainer-framework)

对 Sanaka，很适合的入口任务有：

- 文案统一
- 模板整理
- Windows 打包文档
- 机器格式文档校正
- 小的 UI 一致性修复
- 某个单独平台的兼容性验证

### 3. 主动邀请，不要只等人上门

研究和维护经验都说明，主动邀请很重要。  
参考：
- [How to manage casual contributors to open source projects](https://opensource.com/article/17/10/managing-casual-contributors)

你可以在：

- GitHub issue
- B 站
- 抖音
- 你的个人博客
- 技术群

直接说：

- 现在缺什么角色
- 进来先做什么
- 做到什么程度可以成为长期协作者

### 4. 给成长路径

贡献者留下来，靠的不只是热情，还靠“我接下来能干嘛”。

所以你应该写一个很短的成长路径：

- `Contributor`
- `Trusted Contributor`
- `Maintainer`

每一级对应：

- 权限
- 责任
- 晋升条件

这会比“以后再看”强很多。

## 六、对 Sanaka 最适合的治理模型

我建议你用这个：

### 当前阶段

**Founder-led / BDFL-lite**

规则：

- 你是唯一 owner
- 你保留最终产品和设计否决权
- 主分支合并默认需要你批准
- 贡献者可以广泛参与 issue / PR / 文档 / 测试

### 未来阶段

**Founder-led + delegated maintainers**

规则：

- 你仍然是 owner
- 某些模块可以授权 maintainer
- maintainer 负责日常 review
- 但架构、视觉语言、路线图仍然由你收口

这个模型比一开始就“委员会制”更适合你。

原因很简单：

- Sanaka 还太早
- 产品人格还很强
- 设计方向还没完全固化
- 你本人就是项目辨识度的一部分

2. 新增 `CONTRIBUTING.md`

- 审美由你定
- 主线合并规则由你定

- 每个小 bug 都亲自修
- 每条 issue 都亲自回
- 所有环境问题都亲自排
- 所有文档都自己写到底

- 规则没写

- 项目越来越大，但你还在按“个人仓库”方式管理

所以未来不是你要不要退位，而是你要不要从“唯一执行者”升级成“制度化 owner”。

## 八、如果我被架空了，会怎么样

先说最现实的：

**被架空不一定等于你被“赶出去”，更常见的是你还叫 owner，但项目实际上已经不按你的意志运转。**

会发生的事情通常是这些：

- 你名义上还是创始人
- 但路线图不是你在定
- PR 合并节奏不是你在控制
- 新贡献者先认识的是“活跃维护者”，不是你
- README、release notes、roadmap 的口径慢慢变成别人的
- 你如果反对，别人会觉得你是在“回来夺权”

这其实是开源项目里很典型的“事实权力替代名义权力”。

如果真走到这一步，对你会有三层影响。

### 1. 产品层面

你会最难受，因为项目还在叫 `Sanaka`，但越来越不像你想做的那个 `Sanaka`。

这比项目死掉还难受。  
项目死掉，至少是结束。  
项目活着但变成别人的东西，是持续性消耗。

### 2. 社区层面

别人会默认：

- 现在真正说了算的是那批活跃的人
- 你只是历史人物

如果你这时候再强硬介入，舆论往往不会天然站你这边。很多围观者只看“谁最近在干活”，不看谁最早定义了项目。

### 3. 心理层面

你可能会出现两个很危险的反应：

- 彻底摆烂：那你们玩吧
- 情绪爆炸：开始公开清算所有人

这两种都能理解，但都容易把你自己的位置进一步削弱。

### 真正的风险不是“别人厉害”

真正的风险是：

- 你的所有权没有制度化
- 你没有提前定义角色和权限
- 你没有把核心判断写成项目规则

所以“被架空”的最佳应对，不是在被架空之后狠狠干一仗，而是在那之前把：

- owner 身份
- 决策边界
- 维护者晋升机制
- fork 与品牌规则

全部写清楚。

## 九、如果 fork 之后完全不提我，我想举报、曝光他们，这正当吗

这个问题我分三层说：

- 情绪上
- 社区伦理上
- 法律 / 许可证上

### 1. 情绪上：完全正当

如果一个项目明显建立在你的工作、你的方向、你的原始实现之上，fork 之后却在 README、关于页、发布说明里刻意不提你，甚至努力把你的存在擦掉，你生气、想曝光、想反击，这很正常。

这不是小心眼。

因为对创始人来说，被抹掉的不是一句感谢，而是：

- 贡献历史
- 身份归属
- 名誉资本
- 未来话语权

这个愤怒是合理的。

### 2. 社区伦理上：他们如果刻意抹掉你，通常是很难看的

fork 本身没问题。  
GitHub 和开源世界都默认允许 fork。  
参考：
- [GitHub Docs: Fork a repository](https://docs.github.com/articles/fork-a-repo?platform=windows)

但“可以 fork”不等于“可以把前人的历史当不存在”。

GitHub 的维护者建议里其实反复强调一件事：  
fork 是允许的，分歧也是允许的，但好的维护者会把流程、边界、出处讲清楚，而不是靠暧昧表述偷项目正统性。  
参考：
- [GitHub Open Source Guide: Best Practices for Maintainers](https://github.com/github/opensource.guide/blob/main/_articles/best-practices.md)

所以如果对方是：

- fork 了你的仓库
- 保留了代码主体
- 却在 README / 官网 / about 里完全去历史化

那在社区伦理上，通常是站不太住的。

### 3. 法律 / 许可证上：要看他们抹掉了什么

这里要冷静一点。

如果对方只是：

- README 没提你
- about 没写你
- 但保留了源码里的版权声明
- 保留了许可证
- 没删掉应保留的 legal notice

那**他们很可能在法律上不一定已经违规**，只是做人难看。

GNU/FSF 的文档重点一直是：

- 版权声明要保留
- 许可证声明要保留
- 某些 legal notice / attribution 不能乱删

参考：
- [GNU GPL FAQ](https://www.gnu.org/licenses/gpl-faq.en.html)
- [GNU GPL v3](https://www.gnu.org/licenses/gpl.en.html)
- [How to Use GNU Licenses for Your Own Software](https://www.gnu.org/licenses/gpl-howto.en.html)

尤其 GPL v3 里明确提到可以要求保留合理的 legal notices 或作者归属信息。  
参考：
- [GNU GPL v3, additional terms / author attributions](https://www.gnu.org/licences/gpl.html)

但注意：

**“没有在 README 感谢你”** 和 **“违法删掉版权/许可证通知”** 不是一回事。

### 那你去举报、曝光，正当吗

我的判断：

- 如果他们删了版权头、删了许可证、删了必须保留的 notice
  - 你去主张、投诉、公开指出，正当
- 如果他们只是在人情和项目叙事上故意不提你
  - 你公开表达不满，也可以理解
  - 但更适合“事实陈述式曝光”，不适合纯情绪宣泄式冲塔

也就是说：

**正当，但手法很重要。**

## 十、如果真遇到这种事，怎么做比较有效

我建议按这个顺序。

### 第一层：先保全证据

先存这些：

- fork 时间
- README 改动历史
- about 页面截图
- license / copyright 头是否被删
- commit 历史里哪些核心代码来自你

不要先开骂，先留证据。

### 第二层：先私下发正式说明

发一段短的、冷的、可转发的文字：

- 你承认 fork 的权利
- 你指出他们目前没有正确保留哪些内容
- 如果只是伦理问题，也明确指出“这不符合基本署名惯例”
- 给一个修正期限

这一步的作用是：

- 让你显得站得住
- 给对方一个补救窗口
- 后面真公开时，别人会看到你先讲过道理

### 第三层：如果涉及许可证问题，按许可证路径处理

如果他们删了：

- LICENSE
- 源文件版权头
- 需要保留的 legal notice

那你可以：

- 发 issue
- 发正式邮件
- 必要时发 DMCA / 版权投诉

但这一步前提是你真的有证据，而且最好聚焦在**具体违规点**，不要把“我不爽”包装成“法律事实”。

### 第四层：公开时用“事实帖”，不要用“骂战帖”

最有力的公开写法，不是：

- 他们全是垃圾
- 他们偷了我一切

而是：

1. 项目源自哪里
2. 你做了哪些核心工作
3. fork 之后他们改了什么
4. 哪些署名/notice 被删了
5. 你已经给过修复机会
6. 他们是否拒绝修复

这种写法更容易赢得中立旁观者。

### 第五层：从此以后，把“你的存在”写进制度

如果你真的很在意这个风险，那以后不要只靠道德感。

你应该在 `Sanaka` 里提前加：

- `AUTHORS.md`
- `HISTORY.md`
- `GOVERNANCE.md`
- `TRADEMARK.md`

尤其是：

- 什么叫官方 Sanaka
- 什么叫非官方 fork
- 哪些名称、logo、品牌叙述不能随便用

这样以后别人 fork，可以 fork 代码，但更难 fork 走“正统叙事”。

## 十一、Windows 的 QEMU 前端，除了 Sanaka，还有哪些

我查完之后的结论是：

**不是没有，但真的不多，而且大多数要么老、要么散、要么不成气候。**

### 能找到的几个

#### 1. AQEMU

这是最常被提到的一个。

它是跨平台 QEMU GUI，明确写了支持 Windows 8/10/11。  
参考：
- [AQEMU GitHub](https://github.com/AQEMU/aqemu)
- [AQEMU 官方仓库（历史仓库）](https://github.com/tobimensch/aqemu)

但它的问题也很明显：

- 老项目气质重
- 更像“QEMU 管理器”
- 产品语言和现代桌面应用体验差得远

#### 2. EmuGUI

这是一个个人做的 QEMU GUI。作者自己也写了，是因为不喜欢现有 Windows 上的 QEMU 界面才做的。  
参考：
- [EmuGUI GitHub](https://github.com/Tech-FZ/EmuGUI)

这很能说明问题：

**连做它的人都觉得“Windows 上现有的 QEMU 前端不够好”。**

#### 3. QEMU Manager

这是更老牌的 Windows QEMU 图形前端类型。今天还能搜到不少资料引用它。  
参考：
- [QEMU Manager（资料引用）](https://es.wikipedia.org/wiki/Qemu_Manager)
- [QEMU 词条中提及 Windows GUI 前端](https://de.wikipedia.org/wiki/QEMU)

但它基本属于“历史存在感”，不是现代产品级路线。

#### 4. qqX / Quickemu 相关

这些更接近“帮你更方便地跑 QEMU”的工具链，而不是你现在做的这种面向桌面最终用户的 Windows 原生前端。  
参考：
- [quickemu](https://github.com/quickemu-project/quickemu)
- [qqX](https://qqxproject.org/)

它们有价值，但气质完全不同。

### 所以你的判断其实是对的

如果你说的是：

**“现代一点、产品化一点、能让普通 Windows 用户愿意真正使用的 QEMU 前端”**

那这个空间确实很空。

不是绝对没有人做过。  
但真正做成“像个产品、不是像个项目工具壳”的，非常少。

这也是为什么我觉得你对 `Sanaka` 的嗅觉不差：

- 你没有去做第 100 个 Electron 笔记软件
- 你没有去做另一个聊天壳
- 你是在抓一个“技术上成立，但产品层面长期没人认真收拾”的空档

## 十二、你是不是在做前无古人的东西

严格说，不是“从来没人碰过”。  
但可以说：

**你在做一个“别人碰过，但没有被认真做成现代产品”的方向。**

这已经很难了，也很有价值。

很多所谓蓝海，不是真的“全宇宙没人想到”，而是：

- 有人做过
- 但没人做对
- 或者没人把技术、产品、审美三件事一起做好

Sanaka 如果真能继续往下走，它的稀缺点不是“第一个有 GUI 的 QEMU 前端”，而是：

- 对 Windows / macOS / Linux 统一思考
- 把机器对象、机器包、运行时、显示层、默认保存这些全收束成产品体验
- 把复杂的 QEMU 能力包装成用户能真的用的东西

这个价值，比“是不是世界第一”更重要。
