# GPT -> Kimi

这轮要你接前端正式实现，不是只做设计稿。

我已经把后端能力补上了一部分，现在需要前端把 Builder 高级选项接出来。

## 当前问题

当前仓库里：

- Runtime 已经支持第三方 `pflash` 固件挂载
- 支持用户指定：
  - `firmware.code_path`
  - `firmware.vars_path`
- 支持只给 `CODE`
- 支持同时给 `CODE + VARS`
- `aarch64 + virt` 的 UEFI 后端路线已经开始具备基础能力

但是前端完全没有入口：

- 用户不能选择第三方 `CODE`
- 用户不能选择第三方 `VARS`
- 用户也看不见当前是否在用宿主机自动固件，还是自定义固件

这导致后端能力已经有了，但用户无法操作。

## 目标行为

在创建页 / 编辑页的“高级选项”中，补一组 `UEFI 固件` 配置 UI。

注意：

- 这是高级能力
- 不要做得喧宾夺主
- 但也不能藏得太深，导致根本找不到

用户要能完成这些事：

1. 开/关 UEFI
2. 看到当前 UEFI 是否可用
3. 指定第三方 `CODE` 固件文件
4. 可选指定第三方 `VARS` 固件文件
5. 清除自定义固件路径，回到自动查找模式

## 我这边已经完成的后端能力

你不用管 IPC / Runtime / 文件系统实现，我已经在做/已做：

- 数据模型新增：
  - `advanced.firmware.code_path`
  - `advanced.firmware.vars_path`
- Runtime 会优先使用自定义 `CODE/VARS`
- 如果只给 `CODE`，也允许
- `aarch64 + virt` 允许走 UEFI 固件路线

所以你前端只需要：

- 读写这两个字段
- 把 UI 做清楚

## 你需要做什么

### 1. 在高级选项中增加 “UEFI 固件” 区块

建议位置：

- 就放在现有 `UEFI` 开关下面
- 属于同一组高级启动配置

建议结构：

1. `UEFI` 开关
2. 一段说明文字
3. `固件来源`
   - 自动查找
   - 自定义
4. 如果是自定义：
   - `CODE 固件文件`
   - `VARS 固件文件（可选）`

### 2. 不要暴露内部术语过多，但也别太傻瓜化

可以出现这些词：

- `UEFI 固件`
- `CODE`
- `VARS`
- `pflash`

但不要把界面写成纯工程调试页。

用户文案应该偏：

- “启动固件”
- “自定义固件文件”
- “可选的变量存储文件”

而不是：

- “请手动挂载 pflash block device 0/1”

### 3. 需要的交互

至少支持：

- 选择 `CODE` 文件
- 选择 `VARS` 文件
- 清空 `CODE`
- 清空 `VARS`

如果当前 `CODE` 没填：

- 不要允许用户误以为已经完成了自定义固件配置

### 4. 路径显示方式

路径可能很长。

要求：

- 显示文件名为主
- 次级显示完整路径或截断路径
- 布局不要炸

### 5. 状态文案

需要考虑三种状态：

1. 自动模式
   - 文案类似：
   - `Sanaka 将尝试自动查找可用的 UEFI 固件。`

2. 自定义 `CODE` 已配置，`VARS` 未配置
   - 文案类似：
   - `将使用你指定的固件文件启动。未指定 VARS 文件时，某些固件不会保存启动变量。`

3. 自定义 `CODE + VARS` 都已配置
   - 文案类似：
   - `将使用你指定的 UEFI 固件和变量文件启动。`

### 6. aarch64 的 UI 限制

这点很重要：

- `aarch64` 下，UEFI 不应该继续显示成“不可用”
- 至少对 `virt` 机器类型，前端不能再硬挡住

也就是说：

- 你要检查当前 `supportsUefi` 那套逻辑
- 让 `aarch64 + virt` 也能开 UEFI

## 影响文件

你大概率需要看：

- `src/pages/MachineBuilderPage.tsx`
- 如果要补文案：
  - `src/i18n/resources.ts`

如果你拆组件也可以，但不要为了这点东西过度拆。

完成后至少满足：

2. 可以填写 / 清空：

   - `advanced.firmware.vars_path`
3. 高级选项区不会变成一大坨难看的调试面板
4. 没有把内部 IPC 名、原始 QEMU 参数直接暴露给用户
