Kimi，这次你只做前端，我做后端和主进程。

这轮是“新版本提醒 v1”。

先说清楚：不是自动更新，不是后台下载安装，就是提醒用户有新版本，然后让用户点一下去浏览器下载。

后端和 AppStore 这边我已经接好了，你直接从 `useAppStore()` 拿，不要自己碰主进程。

你现在能直接拿到这些：

- `updateCurrentInfo`
- `updateLastCheck`
- `updateReminder`
- `dismissUpdateReminder()`
- `checkForUpdates(options?)`
- `skipUpdateVersion(version)`
- `openUpdatePage(url)`

解释一下：

- `updateCurrentInfo`
  - 当前版本、当前通道、当前跳过版本
- `updateLastCheck`
  - 最近一次检查结果
- `updateReminder`
  - 一旦后端检查到新版本，这里就会有值
- `dismissUpdateReminder()`
  - 关闭当前提醒
- `checkForUpdates({ silent })`
  - 手动或静默检查
- `skipUpdateVersion(version)`
  - 跳过当前版本
- `openUpdatePage(url)`
  - 用默认浏览器打开下载页

应用行为也定了：

- 启动后静默检查一次
- 运行期间每 8 小时主动检查一次
- 发现新版本就主动提醒
- 设置页里的手动“检查更新”也保留

你前端做这些。

1. 设置页加“更新”区块

里面至少有：

- 当前版本，比如 `0.0.1 (beta)`
- 当前通道，比如 `Beta` / `Release`
- 一个 `检查更新` 按钮

交互要求：

- 点击后要有明确反馈，不能像没点到
- 检查中要有状态
- 已经是最新版本时给轻提示
- 检查失败时给轻提示

注意：

- 不只是手动检查
- 应用本身会主动检查
- 但手动入口必须保留

2. 做新版本提醒弹层

当 `updateReminder` 有值时，弹一个提醒。

这里有两种来源，你前端都要接：

- 用户手动点了 `检查更新`
- 应用启动后或运行中每 8 小时主动检查到新版本

内容至少包括：

- 标题：`发现新版本 0.0.x`
- 如果后端给了 `title`，就显示
- 发布时间 `pubDate`
- 更新内容 `notes`

按钮固定：

- `前往下载`
- `稍后`
- `跳过此版本`

行为：

- `前往下载`
  - 调 `openUpdatePage(url)`
- `稍后`
  - 调 `dismissUpdateReminder()`
- `跳过此版本`
  - 调 `skipUpdateVersion(version)`，然后关掉提醒

3. 更新内容很多时的处理

这点很重要。

更新内容可能很长，所以：

- 内容区单独滚动
- 头部固定
- 底部按钮固定
- 不要整块弹层一起滚
- 不要把弹层撑爆

4. 视觉要求

- 保持和现在 Sanaka 弹层语言一致
- 不要另外发明一套风格
- 不要重边框
- 不要过多粗体
- 重点是清楚、稳定、好读

5. 非常重要：别再出“淡入无动画”那个 bug

你之前出过这种情况：

- 一挂载就直接到最终态
- 没有真正 enter 动画
- 只有 exit 动画
- 鼠标动一下才显示
- 一闪

这次更新弹层不要再这样。

要求：

- 打开时必须真实淡入
- 可以加一点很轻的位移
- 关闭时正常淡出
- 不能依赖鼠标移动才显示
- 不能闪
- 初次挂载不能直接套最终 visible 态

如果你继续用现在的 presence / transition 方案，请你自己先确认 enter 时序没丢。

6. 文案

- 用用户向文案
- 不要把 `.toml`、manifest、metadata、remote config 这种实现细节露给用户

你做完后，回我一份新的 `kimi-want.md`，写清楚：

- 你改了哪些页面和组件
- 长更新内容滚动怎么做的
- 你怎么避免“淡入无动画” bug
- 哪些复用现有组件，哪些是新加的
