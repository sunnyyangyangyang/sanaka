# GPT -> Kimi

这轮只做“网页模式入口”的前端对接，不做后端实现讨论。

参考规范：

- `speccodex/web-mode-entry-v1/spec.md`

## 当前问题

Sanaka 后续会有网页模式，但当前前端没有一个克制、统一、可扩展的入口。

不能再继续按“一个功能一个按钮”往主界面堆：

- 会把页面做成 dashboard
- 顶部/首页越来越乱
- 维护成本会越来越高

另外，设置页也不能承载“现在就执行”的动作。

## 目标行为

前端把“网页模式入口”接到导航栏右侧：

- 主题切换
- `|`
- `...` 更多按钮

点击 `...` 后，菜单里至少出现一项：

- `打开网页模式`

这个入口是网页模式的主入口。

## 你需要做什么

### 1. 在导航栏右侧增加 `...` 更多按钮

要求：

- 放在右侧工具区
- 与主题切换在同一排
- 中间有 `|`
- 视觉上克制，不要喧宾夺主

不要做成：

- 独立的大主按钮
- 首页大卡片
- 新增一排操作按钮

### 2. 在更多菜单中增加 `打开网页模式`

这是这轮唯一必须出现的菜单项。

菜单结构先保持简单，不要为了预留未来能力把它做复杂。

### 3. 留出后端对接位

点击 `打开网页模式` 后，前端这边先按“动作入口”接好。

你需要把这个入口整理成可对接后端调用的结构。

也就是说：

- 不要把它写死成纯展示
- 需要是一个明确的 action handler
- 后续我会接实际的后端能力：
  - 启动网页服务
  - 自动打开浏览器

### 4. 不要把这个动作放进设置页

设置页只允许放冷数据、长期配置、偏好项。

因此：

- `打开网页模式` 不能放设置页
- `现在执行` 类动作都不要塞设置

### 5. 不要默认自动隐藏桌面窗口

这轮只是前端入口规范，不要自作主张加上：

- 自动隐藏桌面窗口
- 自动关闭桌面窗口
- 托盘依赖
- 驻留行为

这些都不属于这轮前端职责。

## 对接约束

### 你这轮只需要完成前端入口与交互壳

后端尚未在这轮正式接入“网页模式服务启动”能力。

所以你只需要把：

- 导航栏右侧入口
- 更多菜单
- `打开网页模式` action

整理好。

后续我会接后端实际执行逻辑。

### 交互方向

最终产品方向是：

1. 用户打开 App
2. 点 `...`
3. 点 `打开网页模式`
4. 系统自动打开默认浏览器

所以前端这轮的结构必须服务于这个流程。

## 影响文件

你大概率会改：

- `src/pages/*`
- `src/components/*`
- `src/i18n/resources.ts`

看你当前导航栏结构在哪。

## 验收点

完成后至少满足：

1. 导航栏右侧存在 `...` 更多按钮
2. 主题切换和 `...` 在同一排，中间有 `|`
3. 更多菜单里有 `打开网页模式`
4. 没有把这个入口做成首页主按钮
5. 没有把这个动作塞进设置页
6. 代码结构上已经留好后端 action 对接位
