# Sanaka Web Mode Entry v1

## 背景

Sanaka 未来可能支持“网页访问本机/远程 Sanaka”的能力，但当前产品仍然以桌面 App 为主。

这个能力如果入口设计不好，会立刻带来几个问题：

- 首页和导航栏不断堆按钮，界面变成 dashboard
- 设置页被塞入大量一次性动作，边界变脏
- 从启动 App 到进入网页模式步骤过多，用户烦躁
- 关闭桌面窗口后，用户不知道如何回到桌面模式

因此需要先定义一套明确、克制、可持续维护的入口规范。

## 目标

本 spec 只定义以下内容：

1. 网页模式入口放在哪里
2. 从启动 App 到进入网页模式的操作上限
3. 设置页与即时动作的边界
4. 桌面窗口与网页模式并存时的默认行为

本 spec 不定义：

- HTTP / WebSocket API 细节
- 认证、安全、远程访问方案
- Runtime / daemon 进程模型
- 托盘、菜单栏常驻等扩展能力

## 产品原则

### 1. Sanaka 仍然是桌面 App 优先

- 默认入口仍然是桌面版
- 网页模式是一个附加能力，不是第二个主产品
- 不能因为支持网页模式，就让桌面主流程变复杂

### 2. 不把首页做成 dashboard

- 不为每个新功能在主工作区堆单独按钮
- 高频主任务保留在主界面
- 附加能力、工具类动作应集中收纳

### 3. 设置页只放冷数据

设置页只允许出现：

- 长期持久化配置
- 用户偏好
- 默认行为
- 需要写入配置文件的内容

设置页不允许出现：

- 一次性即时动作
- 当前会话的操作命令
- “现在就执行”的入口

例子：

- `启动时默认启用网页访问能力`：可以放设置
- `打开网页模式`：不可以放设置
- `停止当前网页服务`：不可以放设置

### 4. 网页模式主流程最多 2 次操作

从用户打开 Sanaka 到进入网页模式，最多允许 2 次手动操作。

如果超过 2 次，这个能力在产品上就已经过重。

## 入口规范

### 导航栏位置

网页模式入口放在顶部导航栏右侧区域，与主题切换同一排。

结构固定为：

- 主题切换
- `|`
- `...` 更多按钮

不单独新增“网页模式主按钮”。

原因：

- 保持主界面克制
- 避免未来每加一个动作就加一个按钮
- 为后续附加能力预留统一收纳位

### 更多菜单

网页模式入口放在 `...` 更多菜单中。

v1 至少包含：

1. `打开网页模式`

后续如果网页服务状态能力补齐，可扩展：

1. `打开网页模式`
2. `复制网页地址`
3. `查看网页服务信息`
4. `停止网页服务`

但 v1 不要求一次性全部实现。

## 交互规范

### 主流程

主流程固定为：

1. 用户打开 `Sanaka.app`
2. 用户点击导航栏右侧 `...`
3. 用户点击 `打开网页模式`
4. 系统自动打开默认浏览器

其中用户手动动作只有 2 次：

1. 点击 `...`
2. 点击 `打开网页模式`

符合“最多 2 次操作”的限制。

### 点击“打开网页模式”后的自动行为

点击后必须自动完成以下动作：

1. 启动网页访问能力（如果尚未启动）
2. 生成当前可访问地址
3. 自动打开默认浏览器
4. 默认进入网页首页，或进入当前上下文对应页面

如果未来能稳定映射当前机器/当前控制台上下文，优先直接进入当前机器对应网页页面。

如果暂时不能稳定映射，则进入网页首页也可接受。

### 不允许的行为

点击“打开网页模式”后，不允许要求用户继续手动：

- 复制地址再自己粘贴到浏览器
- 自己记端口
- 自己查日志
- 自己再点一次“在浏览器打开”

这些都会破坏 2 步目标。

## 桌面窗口默认行为

### v1 默认不自动隐藏桌面窗口

点击“打开网页模式”后：

- 自动打开浏览器
- 桌面窗口继续保留
- 不自动最小化
- 不自动隐藏
- 不自动退出

原因：

- 避免用户误以为 App 消失或退出
- 避免立刻引入“如何回到桌面版”的复杂问题
- 避免把托盘 / 菜单栏常驻变成 v1 必做项

### 提示文案

首次进入网页模式后，可以显示轻提示，但必须克制：

- 已在浏览器中打开网页模式
- 如需节省资源，你可以稍后关闭桌面窗口
- 想回到桌面版时，重新打开 Sanaka.app 即可

这个提示不应打断主流程，不应强制确认。

## 设置页规范

### 可以进入设置的内容

以下内容属于冷数据，可进入设置：

- 是否默认启用网页访问能力
- 默认监听地址
- 默认端口
- 是否允许局域网访问
- 是否启动时自动打开浏览器

### 不可以进入设置的内容

以下内容不允许进入设置：

- 打开网页模式
- 关闭网页模式
- 复制网页地址
- 查看当前会话服务状态

这些都属于即时动作，应留在当前会话的操作入口中。

## 设计约束

### 导航栏负载控制

- 导航栏右侧只允许一个 `...` 收纳入口
- 后续新增工具类动作，应优先放入该菜单
- 不要在顶部右侧不断并排堆按钮

### 不依赖托盘

v1 不应依赖托盘/菜单栏图标作为唯一入口。

原因：

- 跨平台维护复杂
- 用户心智不统一
- 一旦依赖托盘，网页模式与桌面模式的关系会更难理解

后续如果要做托盘，只能作为增强能力，不作为基础能力前提。

## 验收标准

满足以下条件即可视为本 spec 达标：

1. 网页模式入口位于导航栏右侧 `...` 更多菜单中
2. 用户从启动 App 到进入网页模式，手动操作不超过 2 次
3. 点击入口后系统会自动打开默认浏览器
4. 点击入口后桌面窗口默认仍保留
5. 设置页没有承载“打开网页模式”这类即时动作
6. 顶部导航栏没有因为该能力额外新增独立主按钮

## 后续非本期项

以下可以作为后续迭代，但不属于本 spec 本期要求：

- 进入网页模式后自动定位到当前虚拟机控制台
- 网页服务状态面板
- 托盘 / 菜单栏驻留
- 无窗口模式
- 远程连接另一台 Sanaka
- 认证 / token / TLS / 用户权限模型
